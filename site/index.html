01 — what it does

Stop being the message bus for your own agents.

01

One fleet, no new daemon

Coordinate Claude Code, Aider, OpenCode, and Grok as a single fleet. flotilla is drop-in — it talks to the agent sessions already running in tmux, so nothing is replaced and you give nothing up to opt in.

02

A chief of staff that routes & reports

One hub agent — the executive officer (XO) — routes work to the domain agents and collects their replies. You talk to one agent, not five, and manage the whole fleet from your phone.

03

Durable, auditable channel

Every instruction and reply is mirrored to a chat channel under each agent's own identity — a timestamped, phone-readable transcript of all coordination. The audit trail is a first-class feature, not an afterthought.

02 — how it works

Hub and spoke, built on substrate you already have.

  1. delivery & wake

    Inject into a tmux pane

    Each agent lives in a tmux pane. flotilla delivers an instruction by typing it into the target pane — the same thing you do by hand. For a turn-based agent, the injection is the wake; there's nothing to poll.

  2. topology

    One hub routes the rest

    One agent is the hub (the XO). You talk to the hub; it routes to the domain agents and collects their reports — so there is always one coherent picture and one accountable router.

  3. audit & read-back

    Mirror to a chat channel

    Every instruction and reply is posted to a dedicated channel, each agent under its own webhook identity — a durable, timestamped transcript you can read back from any device.

  4. bounded autonomy

    Per-agent permission posture

    Each agent runs with its own allow-list, acting unattended on safe operations while still stopping for confirmation on risky ones. Coordination never implies unbounded authority.

03 — pluggable architecture

Heterogeneous by design. One interface, many harnesses.

Each agent declares a surface that selects a driver — the per-harness policy for how flotilla submits a turn, reads the pane's rendered state (working / idle / awaiting-approval / errored / crashed), and rotates context. A single fleet can mix harnesses behind one interface.

✓ shipped today
  • Surface drivers — Claude Code, Aider, OpenCode, Grok
  • Inter-harness fleets — one XO drives a mixed fleet
  • The self-continuing clock — heartbeat + liveness watchdog
  • Change-detector — wake the XO only on a material change
  • Inbound relay — message the fleet from the channel
◌ designed pluggable · roadmap
  • Modes — first-class behavior modes you drive the fleet by
  • Interfaces — pluggable surfaces; the chat channel is just one
  • Reporting — a richer reporting model beyond the mirror
  • Cursor driver — next harness on the roadmap

These follow the same pluggability model the harness drivers already prove. They're the target, not yet the implementation — built in the open.

05 — 30-second start

Install, send, see your fleet.

  1. 1

    Install

    go install github.com/jim80net/flotilla/cmd/flotilla@latest
  2. 2

    Tag an agent you run, then send

    flotilla register backend --pane demo:0.0
flotilla send --from me backend "pull origin main and run the tests"

    A stable marker survives the TUI retitling its pane; send then confirms the turn actually started.

  3. 3

    Run the self-continuing clock

    flotilla watch --roster ./flotilla.json --ack-file ./flotilla-xo-alive

The full walkthrough — roster, the Discord audit mirror, the self-continuing clock — runs cold from a clean machine.
